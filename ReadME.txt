Description: 

This code was developed as part of a postdoctoral research project focused
on spin excitations in disordered spin systems. The current implementation
performs Monte Carlo (MC) simulations with parallel tempering and semiclas-
sical molecular dynamics (SMD) for J1–J2 Heisenberg and Ising models on
two-dimensional lattice systems;

Available periodic geometries include square, triangular, Lieb, hexagonal,
and Kagome lattices. Other geometries must be configured manually within
the code or loaded at runtime, as in the case of the Ammann–Beenker qua-
sicrystal (QC) approximants provided (see the directory QCrystal_Data);

For periodic systems, a J3 exchange coupling is implemented through the pa-
rameter JX. For the aforementioned QC approximants, this parameter instead
corresponds to a J5 exchange coupling. An external (z-axis) magnetic field
can be included by setting a finite value for the corresponding parameter;

Interaction anisotropy can be introduced by setting the Sz–Sz coupling fac-
tors to realize an XXZ model. A system with disorder due to lattice impuri-
ties can be obtained by setting the disorder ratio (or fraction) parameter
in one of the configuration files (if set to 0, the system is clean);

For SMD simulations, MC–generated spin configurations (samples) recorded at
temperatures below a certain threshold (defined within the code) during the
measurement stage are required. A specific input configuration file then
defines the target sample file;

The main output of the SMD simulations is the averaged dynamical structure
factor (SF). This quantity is recorded for several frequency slices & wave
vectors within the first Brillouin zone, as well as along a predefined path
for varying frequencies;

Additionally, MC and SMD codes employ OpenCV functions to produce images of
sampled spin configurations (including final configurations from both ther-
malization and measurement stages), as well as videos showing system evolu-
tion in MC time and real time. A lattice inspection feature is also imple-
mented using OpenCV, allowing the user to interactively verify the neigh-
bors of each lattice site;

About the model:

Heisenberg (or Ising) Hamiltonian with multiple exchange couplings and ex-
ternal magnetic field for a 2-dimensional (periodic or not) lattice system:

 H = J1 · sum_{<i,j>}     S[i] · S[j] | Nearest neighbors (NN);
   + J2 · sum_{<<i,j>>}   S[i] · S[j] | Next-NN (NNN);
   + JX · sum_{<<<i,j>>>} S[i] · S[j] | Further-NN;
   - hz · sum_i           S[i]_z      | Zeeman term;

 S[i] = ( S[i]_x , S[i]_y , S[i]_z ) : spin vector at site i;

 <i,j>      :  NN site pairs (i,j) sum;
 <<i,j>>    : NNN site pairs (i,j) sum;
 <<<i,j>>>  : Same for pairs of 3rd or 5th (QC case) NN;

 J1, J2, JX : exchange coupling constants (*);
 hz (extH)  : external mag field applied along z-direction;

 *) JX = J3 (crystal case) or J5 (QC case);

> Guide to set configuration files:

The parameters and settings for MC and SMD simulations are defined by 5 in-
put files, which are generated by the Python-code 'Set_Params.py'; This co-
de allows the user to configure the simulations and also prepare the custom
directory, where the compiled MC and SMD codes can be executed & the output
data stay organized in specific folders automatically created. This procedu-
re can be done manually by running the custom shell scripts 'MC_Prep.sh' &
'SMD_Prep.sh' when the input files are already avaiable, these are:

1) Settings_MODEL.txt

Set model parameter: J1, J2 and JX (this is J3, or J5 for QCs), delta[J2]
for NNN d-wave type ALM coupling, NN & NNN Sz-Sz anisotropy factors, exter-
nal magnetic field, the smallest & largest simulation temperatures (number
of MPI threads defines the number of T-grid points), lattice linear size
and geometry (square, triang, hexagn, kagome, lieb);

2) Settings_MC_SIM.txt

Set MC simulation parameters: number of thermalization/measurements sweeps, 
initial spin-state (loww_T, high_T, zpolar, etc), temperature grid function 
(linear, nonlin, custom / see tha 'Make_custom|nonLin_TGrid.py' codes), and
spin-configuration recording skip (this defines the number of samples to be
recorded in the files for SMD simulations);

3) Settings_SMD_SIM.txt

Set SMD simulation parameters: time steps for evolution, temperature list
size (this is the number of the temperature points in the T-grid, equal to
MPI worldSz), input temperature list position (this selects the sample file
for a given T on the grid: 0,1,2, ..., worldSz - 1), time discretization, &
temporal window form (gauss, hannX, lancz, nowin) applied during the tempo-
ral Fourier transform to obtain the spin-spin correlations;

4) Settings_Disorder.txt

Set the disorder ratio (fraction of sites), and the 6 exchange coupling pa-
rameters associated with the interaction between the crystal sites & impu-
rity sites (e.g.: value of J1 for impurity ions of Cr interacting with NN
Fe ions from the 'clean' crystal);

5) Settings_CodeExec.txt

This file controls many code features, so that the user does not need to
compile again each time a different setting is needed for testing:

 1: enable  code feature;
 0: disable code feature;

 ptON     : parallel-tempering;  
 ptAdapt  : parallel-tempering grid-adaptation (*);  
 ptTrack  : parallel-tempering config. exchange tracking;  
 ANNL_ON  : annealing mode (temperatures are reduced gradually);
 FFTW_ON  : FFTW library for discrete Fourier transforms (**);  
 RKCheck  : Runge-Kutta testing code (see the source for details);  
 PBC_OFF  : periodic-boundary-conditions (for periodic geometries);   
 IsiModel : Ising model (replace Heisenberg model in MC);   
 qcrystal : quasicrystal simulations (require other settings);   

 with_recField  spin-configurations (spin-field) recording (SMD-code)
 with_DFTCodes  include DFT codes / structure-factor calculations (MC-code);

 Below, only features for the simulations with the Aamman-Beenker QCs:

 force_Stripes : force stripe-patterms on the Aamman-Beenker approximants;  
 refState_Zero : enable when there is no reference for Neel & Stripe states;
 qct_NeelPhase : enable when the system is known to be in the Neel phase; 
 qct_RemovePBC : enable when using periodic approximants that need PBC off; 
 rec_orderMaps : enable to record local order parameter maps;
 
 (*) Please, avoid enabling this feature since the 'grid-adaptation' code
     is still on testing since the current algorithm is not working well;
     
 (**) Only FFT code from the MKL are being used now, so set this to 0;

> Guide for MC-simulations: (requires Intel's oneAPI with MKL | mpicxx)

1) Run preparion 'MC_Prep' script and follow the instructions: ./MC_Prep.sh 

It creates a custom-named simulation directory inside a working directory
with all output folders, configuration files & scripts automatically managed;

2) Go to the simulation directory '../Work_Dir/HM_SIM(folder_name)/' & run
   the script '0Run_MC_SIM.sh' with the desired number of threads (the num-
   ber of T-grid points is equal to this MPI exec. parameter): 
   
   ./0Run_MC_SIM.sh 24
     
3) For different MC simulations settings, change inputs in MC_SIM; and if
   changes in the MODEL inputs are made, one should usually clean all fol-
   ders to avoid accumulating data and getting erroneous results;
   
   But if changes are made on a specific parameter (J1 , J2 or extH) with
   the option 'Output_Files_Tag' correctly assigned (e.g.: change J1 with
   tag set to 'J1tag'), then the output files will not mix (i.e., accumu-
   late with each consecutive run) and one obtains additional results in
   the 'MC_Collect' folder where the MC averages are recorded for fixed
   temperatures and a varying parameter (only those mentioned);
   
   If the lattice size, geometry or other parameter is changed, or if the
   user wants a fresh simulation, then the script 'MC_Clean.sh' provides
   an automatic way of erasing all content in the output folders & also
   (optinally) in the 'outBin' folder where the lattice properties, bin-
   averages and field samples are recorded;

4) When running MC simulations for disordered systems, one has to run the
   script '0Run_MC_SIM.sh' several times, console example below:
   
   for i in {1..100}; do ./0Run_MC_SIM.sh 24; done
   
   Above, 100 simulations for different (random) disorder configurations
   are executed, each run accumulates the bin-averages and field samples
   into the same files so that the last run outputs correspond to the
   final averages over MC sampled fields & impurity disorder configs.;

> Guide for SMD-simulations: (requires Intel's oneAPI with MKL | mpicxx)

1) Run preparion 'SMD_Prep' script and follow the instructions: ./SMD_Prep.sh 

It looks for an existing custom-named simulation directory inside a working
directory where MC collected samples are avaiable the procedure to execute;

2) Go to the simulation directory '../Work_Dir/HM_SIM(folder_name)/' & run
   the script '0Run_SMD_SIM.sh' with the desired number of threads for pa-
   rallel computing of the temporal evolution & calculations of all samples
   within the target file:
   
   ./0Run_SMD_SIM.sh 4
   
3) For different SMD simulations settings, change inputs in SMD_SIM;

> Outputs: quick guide

The codes for MC and SMD simulations record output data on the directories:
outData (main results, data-files), outFigs (PNG figures - OpenCV), outVids
(AVI videos - OpenCV), outBin (binary files with raw data & files that allow
MC simulations to recover from crash or accumulate data from previous runs
and also sampled spin configurations for SMD simulations);

Quantities averaged during MC runs are recorded in 3 files, where the final
average value and errors are found:

'outData/MC_Stat_Results*_TAG(X).dat';

Here, TAG is a string (J1, J2 or extH) & X is a number (parameter value);

Other results, like static structure factors and additional measurements
performed during MC simulations are recorded in files located in:

'outData/MC_Results/...';

The main output from SMD simulations are the dynamical structure factors,
these are recorded to files that can be found in:

'outData/SMD_Results/...';

When plotting the SF results, using a logarithmic scale helps to reveal
the intensity structures across the Brillouin zone, and defining upper
and lower bounds for the intensity range can significantly improve visuals;

Tip for visualization of long tabular data: disable text wrapping (gedit);
